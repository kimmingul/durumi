# v0.2 Hardening Spec

> **Status (2026-05-14)**: **Complete.** All 11 items shipped across
> v0.2.0 → v0.2.2 plus the post-2.2 refactor + release-gate cycle.
> See the **"v0.2.x hardening complete"** section at the bottom for
> the consolidated ledger. The body of this document remains as the
> original spec + landed-commit annotations for each item.

**Source of truth**: external Codex review of v0.1.14, cross-verified
against current source (May 2026). Every item below was confirmed by
reading the cited file/line before it landed in this spec.

Durumi v0.1.14 has reached feature parity for "Typora-style markdown
editor + medical-research workflow" — the editor, reference manager,
CriticMarkup tracking, AI palette, and export pipeline all ship. The
remaining gaps are not features but **integrity, security, and
maintainability** — the things that decide whether the app is safe to
hand to a clinician who will store real manuscript drafts in it.

This spec groups those gaps into four priority bands and sequences them
so each band can be landed without blocking the next.

---

## Priority bands

- **P0 — Data integrity / trust.** Land in v0.2.0. Failure here loses
  user data or exfiltrates through document handoff.
- **P1 — Security posture.** Land across v0.2.x patches. Required
  before any non-developer distribution.
- **P2 — Maintainability.** Continuous. Done lazily, file-by-file,
  alongside other work — never as a single "big refactor" PR.
- **P3 — Release gates.** One-shot work done immediately before the
  first signed public build.

## P0-1 — Atomic Markdown body write

**Problem.** `electron/ipc.ts` writes the user's primary document with
a direct `fs.writeFile` call:

- [electron/ipc.ts:205](../electron/ipc.ts#L205) — `file:save`
- [electron/ipc.ts:221](../electron/ipc.ts#L221) — `file:saveAs`
- [electron/ipc.ts:245](../electron/ipc.ts#L245) — `export:file` (HTML)

A crash or power loss mid-write leaves the file truncated to whatever
bytes the OS had flushed. Meanwhile the **memo sidecar** at the same
file ([electron/ipc.ts:162-179](../electron/ipc.ts#L162-L179)) and the
**`.bib` writer** at [electron/bibliographyWrite.ts:137](../electron/bibliographyWrite.ts#L137)
already use tmp+rename. The body — the most valuable file in the app —
is the *least* protected. This is the wrong asymmetry.

**Fix.**

1. Extract the existing tmp+rename logic into a shared
   `writeFileAtomic(path, content, encoding?)` in `electron/fs.ts`.
   Same-directory tmp file (`<path>.tmp-<pid>-<ts>`) → `fs.writeFile`
   → `fs.rename` → cleanup on failure. POSIX rename is atomic when
   tmp + target share a directory.
2. Replace all three call sites above with `writeFileAtomic`.
3. Reuse the same helper from `writeMemoSidecar` and
   `bibliographyWrite` to remove the duplicated pattern.

**Out of scope.** Rotating backups (e.g. `<file>.bak`) — separate
concern, debatable UX. fsync after rename — Electron + macOS APFS gives
crash-consistent rename without explicit fsync; Windows NTFS does too
since Vista. Skipping.

**Tests.**

- New `tests/electron/writeFileAtomic.test.ts`: happy path, rename
  failure cleans up tmp, write failure surfaces the original error,
  concurrent writes to the same path don't corrupt the target
  (last-writer-wins is acceptable).
- Update existing `file:save` tests to assert no tmp file remains
  after a successful save.

## P0-2 — CriticMarkup preserve mode HTML escape

**Problem.** The `transformCm(... 'preserve', 'html')` branch
inserts annotation text directly into HTML:

```ts
// shared/criticMarkup.ts:235-243
case 'insert':       return `<ins>${a.text}</ins>`;
case 'delete':       return `<del>${a.text}</del>`;
case 'substitution': return `<del>${a.oldText ?? ''}</del><ins>${a.newText ?? ''}</ins>`;
case 'highlight':    return `<mark class="cm-highlight">${a.text}</mark>`;
case 'comment':      return `<aside class="cm-comment">${a.text}</aside>`;
```

`src/export/renderHtml.ts` already imports `escapeHtml` and applies it
to every other user-controlled string (title, author, TOC text,
KaTeX/Mermaid error messages). CriticMarkup is the only path that
slips raw user input into HTML.

**Threat model.** A co-author's manuscript arrives as `.md`. The
user opens it in Durumi and exports to HTML for review. If the file
contains `{++<script>alert(1)</script>++}`, the preserve-mode export
runs script in any viewer that opens the HTML. Medical research is a
field where **receiving manuscripts from third parties is the core
workflow** — this is not a theoretical attack surface.

`accept` mode is safe because the annotation text is reinserted as
plain Markdown text and re-parsed by markdown-it, which escapes by
default. Only `preserve` mode is affected.

**Fix.**

1. Apply `escapeHtml` (already in `src/export/escapeHtml.ts`) to
   `a.text`, `a.oldText`, `a.newText` inside the preserve+html branch.
2. To avoid a `src/` → `shared/` import (shared cannot depend on src),
   either (a) move `escapeHtml` to `shared/escapeHtml.ts` and re-export
   from `src/export/escapeHtml.ts`, or (b) inline a 6-line copy in
   `shared/criticMarkup.ts`. Pick (a) — escapeHtml is genuinely shared.

**Out of scope.** A full HTML sanitizer (DOMPurify) over the export
output. `markdown-it` is configured with `html: true` by design (Typora
parity) so a sanitizer would change the rendering contract.
CriticMarkup is the *only* path that wraps user input in raw HTML
tags without going through markdown-it's tokenizer — fixing it at the
source is sufficient.

**Tests.**

- Extend `tests/shared/criticMarkup.test.ts`: assert that
  `{++<b>x</b>++}` in preserve+html mode renders
  `<ins>&lt;b&gt;x&lt;/b&gt;</ins>`, not `<ins><b>x</b></ins>`.
- Add a regression test for the substitution shape with both halves
  containing HTML metachars.

## P1-1 — Enable Electron sandbox *(landed in v0.2.x branch alongside image-render fix)*

**Problem.** [electron/main.ts:35](../electron/main.ts#L35) sets
`sandbox: false`. Combined with the 62 IPC channels in
[electron/ipc.ts](../electron/ipc.ts), a renderer XSS escalates to
arbitrary local file read/write through `file:openPath`, `file:save`,
`fs:listDirectory`, etc.

**Discovery findings (2026-05-13).** Static review and a one-shot
build with `sandbox: true` showed this is *not* the multi-PR project
the original Codex review implied:

- `electron/preload.ts` already uses **only** `contextBridge` and
  `ipcRenderer` (plus a type-only import that erases at compile time).
  No `require()`, no `node:*` modules, no `Buffer`, no `process`. This
  is exactly the shape `sandbox: true` requires of a preload script.
- A grep of `src/` (the renderer) for `require\b`, `node:`, and
  `Buffer.` returned only JSDoc comments and CodeMirror AST `node:`
  property accessors. The renderer is already Node-free (which makes
  sense — `nodeIntegration: false` has been set since v0.1.0).
- `pnpm build` with `sandbox: true` flipped in `main.ts` completes
  cleanly with no new warnings.

The "preload split" PR (originally P1-1b) is therefore unnecessary —
the split is already done.

**Revised plan.** A single PR:

1. Flip `sandbox: true` in `main.ts`.
2. Manual smoke test on macOS and Windows: open/save, workspace,
   references, AI chat, Pandoc export, image paste, fs.watch.
3. Add an E2E smoke test that asserts the app launches under
   `sandbox: true` (P3-1 dependency — needs a stable E2E launch
   environment first).
4. Ship in v0.2.x.

**Caveats.** Static analysis cannot prove the absence of a runtime
issue. The remaining risks are:

- A future third-party preload-injected script (e.g. an auto-updater
  hook) would need re-verification against the sandbox contract.
  Currently the preload is fully under our control.
- `saveImage` was checked and already uses `Uint8Array` end-to-end
  (verified in [src/editor/imagePaste.ts:4-6](../src/editor/imagePaste.ts#L4-L6),
  [src/App.tsx:317-325](../src/App.tsx#L317-L325), and the IPC contract
  at [shared/ipc-contract.ts:292-293](../shared/ipc-contract.ts#L292-L293)).
  Sandbox-safe.

**Acceptance.** App boots with `sandbox: true`, full E2E suite green,
no regressions in file ops, references, AI, Pandoc, or export.

**Companion image-render fix.** Smoke-testing v0.2.1 surfaced a
long-standing bug surfaced (not introduced) by the path guard work:
the editor's image widget always set `<img src="...">` to the raw
markdown URL. For pasted images that's a *relative* path like
`assets/img-…png` that the browser resolves against the renderer's
URL (`file:///…/out/renderer/`), not the document's directory. The
image silently failed to load. Bundling the fix with the sandbox
flip is appropriate because both touch the renderer-to-disk pathway
and benefit from one shared manual smoke test.

The fix introduces a custom `durumi-asset://` protocol handled in
main, plus a renderer-side `resolveImageSrc` helper:

- `shared/assetProtocol.ts` — scheme name + `assetUrlFor(absPath)`
  shared between main and renderer.
- `electron/assetProtocol.ts` — `registerAssetProtocolSchemes()`
  declares privileges (`standard`, `secure`, `supportFetchAPI`,
  `stream`) *before* `app.whenReady()`;
  `registerAssetProtocolHandler()` installs the handler. Every
  request runs through the same `isAllowedPath` allowlist as the IPC
  guard, so a renderer that builds
  `durumi-asset:///%2Fetc%2Fpasswd` is rejected with 403.
- `src/editor/docPath.ts` — `setDocPath` effect + `docPathField`
  StateField (same shape as `editMode.ts`). `MarkdownEditor`
  dispatches the effect whenever its `filePath` prop changes.
- `src/utils/resolveImageSrc.ts` — classifies the markdown URL:
  pass-through for `http`/`https`/`data`/`file`/`durumi-asset`/`blob`,
  wrap absolute filesystem paths (POSIX, Windows drive, UNC) in
  `durumi-asset:///`, join relative paths against `dirname(docPath)`
  before wrapping, fall back to the original string when no
  `docPath` is available (new unsaved buffer — the `<img>` will
  fail to load, which is the correct UX).
- `src/editor/decorations/framework.ts` — `decorationPlugin` gains
  an optional `rebuildOn: StateEffectType<unknown>[]` so the image
  widget can declare a dependency on `setDocPath` and re-resolve
  when the file changes (open / save-as).
- `src/editor/decorations/image.ts` — uses `currentDocPath` +
  `resolveImageSrc` at decoration-build time; the `<img>` now gets
  a URL the renderer can actually load.

Works under `sandbox: true` because the renderer never touches the
filesystem directly — main reads on its behalf through the protocol
handler. The handler also benefits from the same workspace
allowlist invariants as the IPC layer.

**Tests.** `tests/utils/resolveImageSrc.test.ts` — 17 cases
covering the URL-classification matrix, doc-path-joined relative
paths, the no-docPath fallback, and a deliberately-malicious
traversal attempt (documented to land at the main-side guard, not
prevented at the renderer).

**Path-guard dir-trust extension.** The first smoke test of the
image fix surfaced a corollary bug in P1-2: the path guard trusted
the *exact* dialog-returned file path, but not its parent
directory. An image freshly written to `<doc_dir>/assets/img-*.png`
got 403'd by the protocol handler because the asset path wasn't in
the session allowlist (only the `.md` was), the doc wasn't always
inside a workspace folder, and recent files were exact-match only.

Fix in `electron/pathGuard.ts`:

- New `sessionAllowedTrees: Set<string>` checked with the same
  `startsWith(root + sep)` prefix match as workspace folders.
- `allowSessionPath(p)` now also adds `dirname(p)` to the tree
  set — so the dialog handlers that already call it (`file:open`,
  `file:saveAs`, `export:file`, `dialog:openFolder`,
  `dialog:pickFile`, `pandoc:pickCustomPath`, `pandoc:import` /
  `pandoc:export`) extend trust to the doc's directory for free.
- New `allowSessionTree(absDir)` for direct registration.
- `file:openPath` and `file:save` (renderer-supplied but already
  validated paths) also call `allowSessionPath` on success, so
  reopening a file via Recent or the file tree trusts its assets.
- New `bootstrapSessionTreesFromRecents()` called once at
  `app.whenReady()` after preferences load — pre-populates the
  trusted-trees set with the parent of every recent file. Cold
  starts now reach a recent doc's assets without the user re-adding
  the folder.

Trust-scope reasoning: opening a document is an *intent signal*
that the user trusts the surrounding directory. Markdown editors
uniformly assume nearby `assets/`, `figs/`, `images/` etc. are
reachable — matching that assumption keeps the guard from blocking
legitimate workflows while still rejecting `/etc/passwd` and any
absolute path a compromised renderer might construct. Symmetric to
workspace folders, with the same `startsWith(root + sep)` precision
so `/foo` does not accidentally trust `/foo-clone/file`.

Extended `tests/electron/pathGuard.test.ts` with 5 new cases:
`allowSessionPath` extends trust to dirname, `allowSessionTree`
directly registers a tree, prefix-sibling escapes are still
rejected from session trees, paths outside the registered tree
fail, and `bootstrapSessionTreesFromRecents` adds recent-file
dirnames.

**CSP whitelist for the custom scheme.** Second smoke test (after the
path-guard dir-trust fix) revealed that the renderer's CSP was
silently blocking every `durumi-asset://` request *before* it reached
the protocol handler. `index.html` had
`img-src 'self' data: file: https:` — no entry for the custom scheme.
Electron 31 enforces the meta-tag CSP on `<img>` loads regardless of
`secure: true` / `supportFetchAPI: true` privileges; with `sandbox: true`
there's no fallback path either.

Fix: add `durumi-asset:` to the `img-src` directive in
[index.html](../index.html). One-line change. The custom-scheme
requests now reach the main-side protocol handler, where the
path-guard allowlist does its job. The protocol handler's defenses
(`isAllowedPath`, MIME-by-extension whitelist) are still the
final authority; CSP just lets the request through the renderer
boundary.

Codex review identified this as the highest-confidence root cause
(direct code evidence of the missing scheme in CSP), separate from
two lower-confidence hypotheses about path-guard ordering after
`image:save` and percent-encoded URL shape — neither of which the
user's scenario actually hit, since `image:save` is itself guarded
by an already-trusted `contextFilePath` and Chromium's standard-scheme
URL parsing handles `%2F` segments stably.

## P1-2 — Path-scoped IPC validation *(landed in v0.2.x branch)*

**Problem.** A compromised renderer could call `file:openPath` with any
absolute path — `/etc/passwd`, `~/.ssh/id_rsa`, etc. Same for every
other path-taking IPC. With ~62 path-touching handlers in `ipc.ts`,
the renderer was effectively trusted to be honest about paths.

**Fix.** New module
[electron/pathGuard.ts](../electron/pathGuard.ts) with three primitives:

- `allowSessionPath(p)` — registers a path as trusted for the rest of
  the session. Called by every main-side dialog handler after it
  resolves.
- `isAllowedPath(p)` / `assertAllowedPath(p)` — returns/throws based
  on three trust sources:
  1. Session allowlist (dialog-returned paths this session).
  2. `prefs.workspaceFolders` — descendants are allowed via
     `resolve()` + `startsWith(root + sep)`.
  3. `prefs.recentFiles` — exact-match only (no directory expansion).
- `assertPrefsPatchAllowed(patch)` — guard on `prefs:set` so the
  renderer cannot inject untrusted paths *into* the allowlist itself
  via `prefs.workspaceFolders` / `prefs.recentFiles`.

Paths are normalised via `path.resolve` before comparison, which
collapses `..` traversal — so `<workspace>/../../etc/passwd` becomes
`/etc/passwd` and fails the workspace-prefix check.

Applied at the handler boundary of every renderer-supplied path:

- **Files**: `file:openPath`, `file:save`, plus all `files:*` mutation
  ops (create, createFolder, rename, duplicate, trash, reveal, index).
- **FS**: `fs:listDirectory`, `fs:watchRoot`.
- **Bibliography**: `bibliography:find`, `ensureFile`, `computePath`,
  `appendEntry`, `readEntries`, `upsertEntry`, `removeEntry`,
  `renameKey`, `importFile`, `autoSaveAbstract`.
- **Reference**: `reference:open` (guards both `bibFilePath` AND the
  joined absolute path to catch `relPath` traversal),
  `reference:extractText` (same dual guard), `reference:status`,
  `reference:scan`, `reference:extractDoi`, `reference:download`.
- **Memo sidecar**: `memoSidecar:read`, `memoSidecar:write`.
- **Pandoc**: `pandoc:export` (sourceFilePath).
- **Other**: `git:getStatus`, `search:workspace`, `image:save` (context
  file path), `prefs:set` (workspaceFolders + recentFiles mutations).

Dialog handlers register their returns into the session allowlist:
`file:open`, `file:saveAs`, `export:file`, `dialog:openFolder`,
`dialog:pickFile`, `pandoc:pickCustomPath`, `pandoc:import` (input
file), `pandoc:export` (output file).

**Symlinks.** `fs.realpath` resolution was intentionally skipped — it
would add an async disk hit to every guard call (hot path) for a
threat (user writes a symlink to `/etc/shadow` *into their own
workspace*) that is exotic enough to accept.

**`pandoc:setCustomPath`.** Not currently guarded. A compromised
renderer can set `prefs.pandocPath` to anything, but the only effect
is that `pandoc:detect`'s subprocess fails to spawn (binary not
executable). Low-impact; documented here so it isn't forgotten.

**Out of scope.** Full zod-based payload validation for every IPC. The
threat is path traversal, not malformed JSON — TypeScript already gives
shape safety on the legitimate caller side. Drop schema validation
for now; revisit only if a non-path payload turns out to be exploitable.

**Tests.**

[tests/electron/pathGuard.test.ts](../tests/electron/pathGuard.test.ts) —
17 cases covering: inside-workspace allowed, exact-match prefix sibling
rejected (e.g. `/foo` workspace does not allow `/foo-clone/file`),
`..` traversal rejected, recent-files exact-match only (not directory
trust), session allowlist with path normalisation, `PathNotAllowedError`
shape, `assertPrefsPatchAllowed` allowing existing entries and dialog-seen
new entries while rejecting unprovenanced ones.

The test module uses an injectable prefs reader
(`_setPrefsReaderForTests`) so it doesn't need an Electron runtime.

## P1-3 — AI key plaintext fallback honesty *(landed in v0.2.x branch)*

**Problem.** [electron/aiKeys.ts](../electron/aiKeys.ts) comments state
the intent is to avoid plaintext fallback, but the code falls back to a
`plain:` prefix when `safeStorage.isEncryptionAvailable()` returns false
(common on Linux without a configured keyring).

**Decision.** Keep the plaintext fallback (blocking AI features on
Linux would gut the product) but surface it to the user. During
implementation the UX was simplified vs. the original sketch:

1. **Up-front warning, no modal.** The original spec called for a
   confirm dialog ("Save plaintext? / Cancel") on every save attempt.
   In practice a *persistent warning paragraph* above the key input —
   visible whenever `aiEncryptionAvailable()` reports false — plus a
   button label change from "Save key" to "Save (plaintext)" is both
   louder and less annoying. The action is self-documenting; no
   one-shot dialog to dismiss.
2. **Per-key status badge.** Below each provider's key input:
   - 🔒 green: "Encrypted in your OS keychain."
   - 🔓 yellow: "Stored in plaintext — no OS keychain available."
   Sourced from a new `aiKeyStatus()` IPC that returns
   `'none' | 'encrypted' | 'plaintext'`. The old boolean `aiHasKey()`
   IPC was renamed/widened — no compat shim.
3. **`AiKeyStatus` type** lives in `shared/ipc-contract.ts` so renderer
   and main share it without a cross-tree import.
4. **Misleading comment fixed.** `aiKeys.ts` header now honestly
   describes both modes and *why* plaintext fallback exists (gutting AI
   on un-configured Linux is worse than a loud plaintext indicator).

**Out of scope.** Bundling a key derivation function (PBKDF2) with a
user passphrase. That is a real product feature and belongs on the
Roadmap, not in hardening.

## P1-4 — `bindToDocument` no longer creates files *(landed in v0.2.x branch)*

**Problem.** `bibliographyStore.bindToDocument` claimed to "record the
would-be path" but actually called `bibliographyEnsureFile`, which
*created* the `.bib` on disk. Opening a manuscript silently wrote a new
file to the user's folder — a UX surprise plus a (small) data risk if
the user closed the tab thinking nothing happened.

**Fix.**

1. Added `computeBibPath(docPath, dirOverride?)` in
   [electron/bibliographyWrite.ts](../electron/bibliographyWrite.ts) —
   pure path lookup that probes the discovery order and returns
   `{ path, exists }` without touching disk if nothing matches.
   `ensureBibFile` was refactored to delegate the probe to this helper
   and only do the write when `exists: false`.
2. New IPC `bibliography:computePath` exposed via
   [shared/ipc-contract.ts](../shared/ipc-contract.ts) and
   [electron/preload.ts](../electron/preload.ts).
3. [src/store/bibliographyStore.ts](../src/store/bibliographyStore.ts)
   `bindToDocument` now calls `bibliographyComputePath` for the
   not-found case and stores `exists: probe.exists`. No disk write.
4. The `.bib` materialises on the first `bibliographyAppendEntry` call
   — its atomic tmp+rename write already creates the file when missing.
   `afterEntryAdded` flips the store's `exists` flag to true (idempotent);
   `mergeImportedEntries` and `registerOrphan` do the same in their
   own write branches.

The "will be created at <path>" UI placeholder was already wired in
[src/components/sidebar/ReferencesTab.tsx](../src/components/sidebar/ReferencesTab.tsx)
via the `references.target.willCreate` i18n key — the existing UI
branch was effectively dead code under the old behavior (because
`exists` was always flipped to `true` on bind). It's now the real
state for un-bound documents.

**Tests.**

- `tests/electron/bibliographyWrite.test.ts` — three new `computeBibPath`
  cases: existing file, default path with side-effect guard
  (`readFile` on the default path must throw — no silent creation),
  null docPath.
- `tests/store/bibliographyStore.test.ts` — the old "falls back to
  ensureFile (creates)" test was renamed to "falls back to computePath
  (no side effect)" with an explicit
  `expect(api.bibliographyEnsureFile).not.toHaveBeenCalled()` regression
  guard. A new test verifies `exists` flips to `true` after the first
  `addEntry` succeeds.
- `tests/sidebar/rightSidebar.test.tsx` — mock updated with the new
  `bibliographyComputePath` handler.

**Out of scope.** A full E2E test that asserts no `.bib` file exists in
the workspace after a document open. E2E coverage is currently flaky
(see P3-1) — the unit-level mock + regression guard is enough until E2E
is stabilized.

## P2-1 — App.tsx decomposition *(landed → [`fd66ff1`](https://github.com/kimmingul/durumi/commit/fd66ff1))*

**Problem.** [src/App.tsx](../src/App.tsx) was 906 lines. It owned menu
command dispatch, file save/open coordination, export/import flow, AI
palette state, citation insert flow, workspace state, and dialog
state — every feature added to Durumi enlarged it.

**Approach.** Single pure-refactor PR that lifted 12 focused hooks into
`src/hooks/`. App.tsx now reads top-to-bottom as orchestration only;
the giant `onMenuCommand` switch became `useMenuCommandRouter` and the
five feature hooks it dispatches to.

Hooks extracted (final landed list):

- `useFileMenuCommands` — doSave / maybeDiscard / new / open / saveAs /
  openPath / loadTemplate. Sidebar `onOpenFile`/`onOpenHit` and
  QuickOpen `onPick` now share the dirty-discard guard via
  `fileCommands.doOpenPath`.
- `useExportFlow` — HTML / PDF render, Pandoc DOCX / LaTeX export,
  Pandoc DOCX import, plus the `pandocInstallOp` state machine so the
  install dialog can retry the original operation.
- `useCitationInsertFlow` — citation dialog / palette / bulk-DOI /
  import-references / AI-cite-suggest state, plus
  `insertCitationAtCaret` and post-rename atomic
  `migrateCitationsInDoc`.
- `useAiPalette` — AI Command Palette state (selection / paragraph /
  from / to / hasKey) with `open` / `close` / `accept` actions.
- `useWorkspaceMenu` — open/close folder + prefs persistence + fs
  watch + git status poll.
- `useMenuCommandRouter` — the giant `window.api.onMenuCommand` switch;
  stitches the five feature hooks above into one dispatcher.
- `useAppChromeEffects` — data-theme attribute mirror, macros
  load+subscribe, system theme broadcast, git-status invalidation,
  window title.
- `usePreferencesInit` — startup fan-out of persisted prefs into the
  six store slices plus journal-preset injection.
- `useCustomCss` — `<style id="custom-css">` upsert + live subscription.
- `useAppCloseGuard` — onAppRequestClose dirty-buffer prompt.
- `useMemoEvents` — sidecar reload + bibliography bind + manually-hidden
  reset + orphan pruning + `durumi:memo-focus` /
  `durumi:memo-panel-toggle` / `durumi:reference-open` DOM listeners.
- `usePickAndInsertImage` — Toolbar Image button → OS picker →
  `saveImage` IPC → `![](relPath)` at caret.

**Acceptance achieved.** App.tsx 246 lines (target was ≤ 400). All
quality gates green: 1308 tests unchanged.

**Non-obvious decisions.** `useExportFlow` takes `maybeDiscard` as a
parameter (not internalised) because the Pandoc DOCX import is the
only export that needs the dirty guard; keeping the dependency
explicit keeps the export hook free of `appStore` mutation
responsibility. `useCitationInsertFlow` and `useAiPalette` take
`editorViewRef` by ref (not value) so async insert/accept callbacks
always operate on the live view.

## P2-2 — `electron/ipc.ts` decomposition *(landed → [`45830b2`](https://github.com/kimmingul/durumi/commit/45830b2))*

**Problem.** [electron/ipc.ts](../electron/ipc.ts) had grown to 889
lines across 8 unrelated domains.

**Approach.** Behaviour-preserving split into 10 per-domain modules
under `electron/ipc/` plus a barrel and a `_shared.ts` for
cross-domain helpers. All `assertAllowedPath` / `allowSessionPath` /
`assertPrefsPatchAllowed` call sites and their ordering preserved
verbatim. `KeyVault` was lifted to the barrel and injected into
`registerAiHandlers` rather than created module-locally so tests can
substitute it.

Landed module shape (all <200 lines):

| File | Lines | Domain |

## P3-1 — E2E launch environment stabilization *(landed → [`c9df4d0`](https://github.com/kimmingul/durumi/commit/c9df4d0))*

**Problem.** Baseline was 7 passed / 9 failed. Two failure clusters:

1. **WYSIWYG strict-literal default broke 7 specs** that typed
   markdown source and asserted live-preview decorations. The v0.1.12
   transactionFilter (shipped as default in v0.1.13) rewrites every
   user-typed special character (`#`, `*`, backtick, `~`, etc.) with
   a leading backslash so the parser sees literal text — heading /
   fence / strikethrough / task nodes never emit, the tested
   decorations never fire. The specs were written against the legacy
   Typora-mode behaviour and silently broke when WYSIWYG became
   default.
2. **Path-guard tmp-bypass + sidebar state pollution** broke 2
   workspace specs. `prefs:set` rejected `os.tmpdir()`-rooted folders
   (no dialog provenance), and persistent prefs from prior runs left
   `sidebar.activeTab = 'outline'` so the file tree wasn't visible.

**Fix.**

- New `e2e/_helpers.ts::setTyporaMode(app, page)` dispatches the
  View ▸ Mode ▸ Live menu command via `menu:command` before typing.
  This is real user-facing UI exercise (same code path as the View
  menu), not a test-only API. Eight affected specs (b1 task list,
  b1 strikethrough, b1 fenced highlight, b2 outline, b3 HTML export,
  c5 mermaid, golden live-preview, b2 open-folder-click) now call
  it after launch.
- `electron/pathGuard.ts`: `DURUMI_E2E=1` env flag (forwarded by the
  `test:e2e` script) accepts any path under `os.tmpdir()` as a
  session-trusted tree. Renderers cannot set main-process env vars,
  so this is opt-in via the test script and does **not** widen the
  production attack surface. Documented inline.
- Both workspace specs now pass `sidebar: { activeTab: 'files', ... }`
  in the same `prefsSet` call so persisted state can't hide the file
  tree.

**Acceptance achieved.** **16 / 16 e2e tests pass** under
`pnpm test:e2e`. No production-code behaviour change other than the
documented `DURUMI_E2E` env bypass.

### What is true of the v0.2.x build

- **Sandbox.** `webPreferences.sandbox: true` on the BrowserWindow.
  Combined with `contextIsolation: true`, `nodeIntegration: false`,
  and the path-scoped IPC guard, the renderer is defence-in-depth
  isolated from the filesystem. Production DMG (macOS arm64/x64) and
  EXE (Windows x64) both build and run cleanly under the sandbox.
- **Path guard.** Every renderer-supplied path goes through
  `assertAllowedPath` before any disk operation. Workspace folders,
  recent files (exact match), session-trusted directory trees
  (descendant prefix), and dialog-returned paths are the only sources
  of trust. Image-paste asset directories are reached via the
  `allowSessionPath → dirname → sessionAllowedTrees` extension.
- **Asset protocol.** `durumi-asset://x/?p=<encoded-abs-path>` serves
  workspace-local images (and any future media) through a
  path-guarded handler in main. Renderer never touches disk
  directly. See [`docs/image-rendering.md`](image-rendering.md) for
  the full case study.
- **Data integrity.** Markdown body, memo sidecar, and `.bib` writes
  all go through `writeFileAtomic` (tmp + rename). CriticMarkup
  preserve-mode HTML output escapes annotation text.
- **AI keys.** OS keychain when available, honest plaintext fallback
  with a per-key 🔒/🔓 badge when not.
- **Bibliography.** No silent `.bib` creation on bind; the file
  materialises only on first `addEntry`.
- **Build.** Renderer main chunk 1.74 MB. Lazy boundaries on
  KaTeX, markdown-it, node-emoji, js-yaml, and all dialog
  components.
- **CI.** 1308 vitest tests + 16 Playwright Electron e2e tests pass
  on every commit. `pnpm lint`, `pnpm typecheck`, `pnpm build` are
  release gates.
- **Release signing.** Runbook + dormant `electron-builder.yml`
  template land the path to a signed dual-platform DMG/EXE; activation
  is gated on the user provisioning paid certs ([`docs/RELEASE.md`](RELEASE.md)).

# v0.2 Hardening Spec

> **Status (planning, 2026-05-13)**: Not started. This document captures
> the hardening roadmap derived from an external code review of v0.1.14.
> P0 items target v0.2.0 (next minor); P1 items target v0.2.x; P2 items
> are continuous; P3 items gate the first public release.

**Source of truth**: external Codex review of v0.1.14, cross-verified
against current source (May 2026). Every item below was confirmed by
reading the cited file/line before it landed in this spec.

Durumi v0.1.14 has reached feature parity for "Typora-style markdown
editor + medical-research workflow" ‚Äî the editor, reference manager,
CriticMarkup tracking, AI palette, and export pipeline all ship. The
remaining gaps are not features but **integrity, security, and
maintainability** ‚Äî the things that decide whether the app is safe to
hand to a clinician who will store real manuscript drafts in it.

This spec groups those gaps into four priority bands and sequences them
so each band can be landed without blocking the next.

---

## Priority bands

- **P0 ‚Äî Data integrity / trust.** Land in v0.2.0. Failure here loses
  user data or exfiltrates through document handoff.
- **P1 ‚Äî Security posture.** Land across v0.2.x patches. Required
  before any non-developer distribution.
- **P2 ‚Äî Maintainability.** Continuous. Done lazily, file-by-file,
  alongside other work ‚Äî never as a single "big refactor" PR.
- **P3 ‚Äî Release gates.** One-shot work done immediately before the
  first signed public build.

## P0-1 ‚Äî Atomic Markdown body write

**Problem.** `electron/ipc.ts` writes the user's primary document with
a direct `fs.writeFile` call:

- [electron/ipc.ts:205](../electron/ipc.ts#L205) ‚Äî `file:save`
- [electron/ipc.ts:221](../electron/ipc.ts#L221) ‚Äî `file:saveAs`
- [electron/ipc.ts:245](../electron/ipc.ts#L245) ‚Äî `export:file` (HTML)

A crash or power loss mid-write leaves the file truncated to whatever
bytes the OS had flushed. Meanwhile the **memo sidecar** at the same
file ([electron/ipc.ts:162-179](../electron/ipc.ts#L162-L179)) and the
**`.bib` writer** at [electron/bibliographyWrite.ts:137](../electron/bibliographyWrite.ts#L137)
already use tmp+rename. The body ‚Äî the most valuable file in the app ‚Äî
is the *least* protected. This is the wrong asymmetry.

**Fix.**

1. Extract the existing tmp+rename logic into a shared
   `writeFileAtomic(path, content, encoding?)` in `electron/fs.ts`.
   Same-directory tmp file (`<path>.tmp-<pid>-<ts>`) ‚Üí `fs.writeFile`
   ‚Üí `fs.rename` ‚Üí cleanup on failure. POSIX rename is atomic when
   tmp + target share a directory.
2. Replace all three call sites above with `writeFileAtomic`.
3. Reuse the same helper from `writeMemoSidecar` and
   `bibliographyWrite` to remove the duplicated pattern.

**Out of scope.** Rotating backups (e.g. `<file>.bak`) ‚Äî separate
concern, debatable UX. fsync after rename ‚Äî Electron + macOS APFS gives
crash-consistent rename without explicit fsync; Windows NTFS does too
since Vista. Skipping.

**Tests.**

- New `tests/electron/writeFileAtomic.test.ts`: happy path, rename
  failure cleans up tmp, write failure surfaces the original error,
  concurrent writes to the same path don't corrupt the target
  (last-writer-wins is acceptable).
- Update existing `file:save` tests to assert no tmp file remains
  after a successful save.

## P0-2 ‚Äî CriticMarkup preserve mode HTML escape

**Problem.** The `transformCm(... 'preserve', 'html')` branch
inserts annotation text directly into HTML:

```ts
// shared/criticMarkup.ts:235-243
case 'insert':       return `<ins>${a.text}</ins>`;
case 'delete':       return `<del>${a.text}</del>`;
case 'substitution': return `<del>${a.oldText ?? ''}</del><ins>${a.newText ?? ''}</ins>`;
case 'highlight':    return `<mark class="cm-highlight">${a.text}</mark>`;
case 'comment':      return `<aside class="cm-comment">${a.text}</aside>`;
```

`src/export/renderHtml.ts` already imports `escapeHtml` and applies it
to every other user-controlled string (title, author, TOC text,
KaTeX/Mermaid error messages). CriticMarkup is the only path that
slips raw user input into HTML.

**Threat model.** A co-author's manuscript arrives as `.md`. The
user opens it in Durumi and exports to HTML for review. If the file
contains `{++<script>alert(1)</script>++}`, the preserve-mode export
runs script in any viewer that opens the HTML. Medical research is a
field where **receiving manuscripts from third parties is the core
workflow** ‚Äî this is not a theoretical attack surface.

`accept` mode is safe because the annotation text is reinserted as
plain Markdown text and re-parsed by markdown-it, which escapes by
default. Only `preserve` mode is affected.

**Fix.**

1. Apply `escapeHtml` (already in `src/export/escapeHtml.ts`) to
   `a.text`, `a.oldText`, `a.newText` inside the preserve+html branch.
2. To avoid a `src/` ‚Üí `shared/` import (shared cannot depend on src),
   either (a) move `escapeHtml` to `shared/escapeHtml.ts` and re-export
   from `src/export/escapeHtml.ts`, or (b) inline a 6-line copy in
   `shared/criticMarkup.ts`. Pick (a) ‚Äî escapeHtml is genuinely shared.

**Out of scope.** A full HTML sanitizer (DOMPurify) over the export
output. `markdown-it` is configured with `html: true` by design (Typora
parity) so a sanitizer would change the rendering contract.
CriticMarkup is the *only* path that wraps user input in raw HTML
tags without going through markdown-it's tokenizer ‚Äî fixing it at the
source is sufficient.

**Tests.**

- Extend `tests/shared/criticMarkup.test.ts`: assert that
  `{++<b>x</b>++}` in preserve+html mode renders
  `<ins>&lt;b&gt;x&lt;/b&gt;</ins>`, not `<ins><b>x</b></ins>`.
- Add a regression test for the substitution shape with both halves
  containing HTML metachars.

## P1-1 ‚Äî Enable Electron sandbox

**Problem.** [electron/main.ts:35](../electron/main.ts#L35) sets
`sandbox: false`. Combined with the 62 IPC channels in
[electron/ipc.ts](../electron/ipc.ts), a renderer XSS escalates to
arbitrary local file read/write through `file:openPath`, `file:save`,
`fs:listDirectory`, etc.

**Discovery findings (2026-05-13).** Static review and a one-shot
build with `sandbox: true` showed this is *not* the multi-PR project
the original Codex review implied:

- `electron/preload.ts` already uses **only** `contextBridge` and
  `ipcRenderer` (plus a type-only import that erases at compile time).
  No `require()`, no `node:*` modules, no `Buffer`, no `process`. This
  is exactly the shape `sandbox: true` requires of a preload script.
- A grep of `src/` (the renderer) for `require\b`, `node:`, and
  `Buffer.` returned only JSDoc comments and CodeMirror AST `node:`
  property accessors. The renderer is already Node-free (which makes
  sense ‚Äî `nodeIntegration: false` has been set since v0.1.0).
- `pnpm build` with `sandbox: true` flipped in `main.ts` completes
  cleanly with no new warnings.

The "preload split" PR (originally P1-1b) is therefore unnecessary ‚Äî
the split is already done.

**Revised plan.** A single PR:

1. Flip `sandbox: true` in `main.ts`.
2. Manual smoke test on macOS and Windows: open/save, workspace,
   references, AI chat, Pandoc export, image paste, fs.watch.
3. Add an E2E smoke test that asserts the app launches under
   `sandbox: true` (P3-1 dependency ‚Äî needs a stable E2E launch
   environment first).
4. Ship in v0.2.x.

**Caveats.** Static analysis cannot prove the absence of a runtime
issue. The remaining risks are:

- A future third-party preload-injected script (e.g. an auto-updater
  hook) would need re-verification against the sandbox contract.
  Currently the preload is fully under our control.
- `saveImage` was checked and already uses `Uint8Array` end-to-end
  (verified in [src/editor/imagePaste.ts:4-6](../src/editor/imagePaste.ts#L4-L6),
  [src/App.tsx:317-325](../src/App.tsx#L317-L325), and the IPC contract
  at [shared/ipc-contract.ts:292-293](../shared/ipc-contract.ts#L292-L293)).
  Sandbox-safe.

**Acceptance.** App boots with `sandbox: true`, full E2E suite green,
no regressions in file ops, references, AI, Pandoc, or export.

## P1-2 ‚Äî Path-scoped IPC validation

**Problem.** A compromised renderer can call `file:openPath` with an
arbitrary string. There is no check that the path belongs to the user's
workspace or to a previously-opened file.

**Fix.**

1. Add `electron/pathGuard.ts` with `assertAllowedPath(path, kind)`:
   - `kind: 'read'` ‚Üí path must be inside a workspace folder OR equal
     to the path returned by the most recent file/open dialog.
   - `kind: 'write'` ‚Üí path must be inside a workspace folder OR equal
     to a dialog-returned path from this session.
2. Apply at the *handler boundary* in `ipc.ts` for: `file:openPath`,
   `file:save`, `file:saveAs` (target dir only), `fs:listDirectory`,
   `fs:readFile`, all `bibliography:*` operations that take a file path,
   `reference:*` operations that take a file path.
3. Maintain a `sessionAllowlist: Set<string>` populated by dialog
   handlers (open/save) ‚Äî entries persist for the session only.

**Out of scope.** Full zod-based payload validation for every IPC. The
threat is path traversal, not malformed JSON ‚Äî TypeScript already gives
us shape safety on the legitimate caller side. Drop schema validation
for now; revisit only if a non-path payload turns out to be exploitable.

**Tests.**

- `tests/electron/pathGuard.test.ts`: workspace-inside path passes,
  arbitrary path fails, symlink escape fails (resolve with
  `fs.realpath` before the check), dialog-returned path passes after
  the dialog runs.

## P1-3 ‚Äî AI key plaintext fallback honesty *(landed in v0.2.x branch)*

**Problem.** [electron/aiKeys.ts](../electron/aiKeys.ts) comments state
the intent is to avoid plaintext fallback, but the code falls back to a
`plain:` prefix when `safeStorage.isEncryptionAvailable()` returns false
(common on Linux without a configured keyring).

**Decision.** Keep the plaintext fallback (blocking AI features on
Linux would gut the product) but surface it to the user. During
implementation the UX was simplified vs. the original sketch:

1. **Up-front warning, no modal.** The original spec called for a
   confirm dialog ("Save plaintext? / Cancel") on every save attempt.
   In practice a *persistent warning paragraph* above the key input ‚Äî
   visible whenever `aiEncryptionAvailable()` reports false ‚Äî plus a
   button label change from "Save key" to "Save (plaintext)" is both
   louder and less annoying. The action is self-documenting; no
   one-shot dialog to dismiss.
2. **Per-key status badge.** Below each provider's key input:
   - üîí green: "Encrypted in your OS keychain."
   - üîì yellow: "Stored in plaintext ‚Äî no OS keychain available."
   Sourced from a new `aiKeyStatus()` IPC that returns
   `'none' | 'encrypted' | 'plaintext'`. The old boolean `aiHasKey()`
   IPC was renamed/widened ‚Äî no compat shim.
3. **`AiKeyStatus` type** lives in `shared/ipc-contract.ts` so renderer
   and main share it without a cross-tree import.
4. **Misleading comment fixed.** `aiKeys.ts` header now honestly
   describes both modes and *why* plaintext fallback exists (gutting AI
   on un-configured Linux is worse than a loud plaintext indicator).

**Out of scope.** Bundling a key derivation function (PBKDF2) with a
user passphrase. That is a real product feature and belongs on the
Roadmap, not in hardening.

## P1-4 ‚Äî `bindToDocument` no longer creates files

**Problem.**
[src/store/bibliographyStore.ts:220-228](../src/store/bibliographyStore.ts#L220-L228)
comments say "record the would-be path" but actually calls
`bibliographyEnsureFile` which creates the `.bib` on disk. Opening a
manuscript silently writes a new file to the user's folder.

**Fix.**

1. Add `bibliographyComputePath(docPath)` IPC: returns the path
   `ensureBibFile` *would* create, without touching disk.
2. `bindToDocument` calls the new IPC and stores `exists: false,
   filePath: <computed>` in the Zustand store.
3. `bibliographyEnsureFile` is invoked only on the first
   `appendEntry` call. UI shows "Bibliography will be created at <path>
   on first reference" until then.

**Tests.**

- `tests/store/bibliographyStore.test.ts`: opening a document does not
  produce any `bibliographyEnsureFile` IPC call.
- E2E: open a markdown file in a workspace, assert no new `.bib` file
  exists in the workspace after the open completes.

## P2-1 ‚Äî App.tsx decomposition

**Problem.** [src/App.tsx](../src/App.tsx) is 906 lines. It owns menu
command dispatch, file save/open coordination, export/import flow, AI
palette state, citation insert flow, workspace state, and dialog
state ‚Äî every feature added to Durumi enlarges it.

**Approach. Never as one PR.** Each future feature that touches App.tsx
extracts the slice it touches:

- `useFileMenuCommands` ‚Äî open/save/close + dirty-close dialog
- `useExportFlow` ‚Äî HTML/PDF export, format selection, recent paths
- `useAiPalette` ‚Äî AI command palette open/close + draft state
- `useCitationInsertFlow` ‚Äî citation picker + insert at cursor
- `useWorkspaceMenu` ‚Äî open folder, close folder, recent folders

**Acceptance.** App.tsx ‚â§ 400 lines after the last hook is extracted.
No deadline ‚Äî opportunistic.

## P2-2 ‚Äî `electron/ipc.ts` decomposition

**Problem.** [electron/ipc.ts](../electron/ipc.ts) is 787 lines across
8 unrelated domains. Same approach as P2-1 ‚Äî feature-by-feature
extraction into `electron/ipc/<domain>.ts` files:

- `ipc/files.ts` ‚Äî open/save/saveAs/openPath/readFile/listDirectory
- `ipc/preferences.ts` ‚Äî prefsGet/prefsSet
- `ipc/search.ts` ‚Äî workspaceSearch/searchInFiles
- `ipc/bibliography.ts` ‚Äî all `bibliography:*`
- `ipc/reference.ts` ‚Äî all `reference:*`
- `ipc/ai.ts` ‚Äî `ai:*` + `aiKeys:*`
- `ipc/pandoc.ts` ‚Äî `pandoc:*`
- `ipc/shell.ts` ‚Äî `shell:openExternal`, `shell:showItemInFolder`

`registerIpcHandlers` becomes a barrel that calls each module's
`register‚Ä¶Handlers`.

**Acceptance.** No file in `electron/ipc/` exceeds 200 lines.
Opportunistic, no deadline.

## P3-1 ‚Äî E2E launch environment stabilization

**Problem.** Codex hit Electron `SIGABRT` running `pnpm test:e2e` due
to `ELECTRON_RUN_AS_NODE=1` leaking in from the shell. Even with the
variable cleared, Electron launch was flaky.

**Fix.**

1. The `test:e2e` script already prefixes `ELECTRON_RUN_AS_NODE=` to
   clear the inherited value. Confirm Playwright `launch.env` also
   unsets it explicitly (belt-and-braces against shell quirks).
2. Add a single Electron smoke test in `e2e/smoke.spec.ts` that just
   launches the app, asserts the main window opens, and exits. Run
   this in GitHub Actions on macOS and Ubuntu runners on every PR ‚Äî if
   launch breaks, the failure is loud and the rest of the suite is
   not blamed.
3. Document the E2E launch contract in `CONTRIBUTING.md`: `pnpm build`
   before `pnpm test:e2e`, the `ELECTRON_RUN_AS_NODE` requirement, the
   smoke test's role.

## P3-2 ‚Äî Bundle size + lazy-loaded chunk audit

**Problem.** Codex reports the renderer main chunk is ~3.06 MB with a
build warning, and the Mermaid / Cytoscape lazy chunks are large.

**Fix.**

1. Run `pnpm build` with `--mode production` and `vite-bundle-analyzer`
   one-off; identify the top 5 contributors to the main chunk.
2. Anything that is not on the editor's critical path moves behind a
   `React.lazy` boundary: Mermaid (already lazy in editor ‚Äî verify it
   also is in export), Cytoscape, KaTeX (export only), PDF.js text
   layer, AI client.
3. Target: main chunk ‚â§ 1.5 MB. Lazy chunks may be large; that's fine
   as long as they load only when triggered.

**Problem.** [electron-builder.yml:19](../electron-builder.yml#L19) has

**Fix.** Tracked separately as Roadmap item 7 in MEMORY. Pre-public-release
work. Not expanded here because the path is well-understood

## Sequence

```
v0.2.0 release branch:
  P0-1 (atomic write)
  P0-2 (CriticMarkup escape)
  ‚Üí ship v0.2.0

v0.2.x patches (interleaved with feature work):
  P1-3 (AI key honesty)         ‚Üê cheap, no design risk
  P1-4 (bindToDocument)         ‚Üê cheap, no design risk
  P1-1  (flip sandbox: true)    ‚Üê preload already sandbox-clean; manual smoke test only
  P1-2  (path guard)            ‚Üê defence in depth even with sandbox on

P2 (continuous, opportunistic):
  Whenever a feature touches App.tsx or ipc.ts, extract the slice.

P3 (gate the first public release):
  P3-1 ‚Üí P3-2 ‚Üí P3-3
```

## Out-of-scope clarifications

- **Markdown rotating backups** ‚Äî debatable UX; not in v0.2.
- **zod for all IPC payloads** ‚Äî TypeScript already handles shape;
  the real threat is paths. Skipped.
- **DOMPurify on export** ‚Äî would break `html: true` markdown-it
  contract. Fix CriticMarkup at the source instead.
- **Bundling a passphrase-derived key for AI keys** ‚Äî product
  feature, not hardening.
- **Worker thread for `search.ts` / `fileIndex.ts`** ‚Äî current
  cancellation + cap is sufficient for medical-research-scale
  folders. Revisit only if a user reports lag.

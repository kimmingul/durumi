## 2. New gaps discovered during v0.2.8–.11 work

These were NOT in the Phase A matrix; they surfaced during implementation/review. Each is a Phase A blind spot worth recording for future audits.

1. **`setEditMode` listener latent bug across 6 fields** (codex review,
   v0.2.8). Phase A only saw `comment.ts` / `criticMarkup.ts` as Doc-
   mode-leaky. The codex follow-up exposed that `citation`, `footnote`,
   `frontMatter`, `math` (blockMathField) and the two repaired fields
   ALL shared a `tr.docChanged || tr.selection` short-circuit that left
   them stale on a bare `Cmd+1` mode switch. **Lesson**: Phase A read
   each decoration's `buildDecorations()` but didn't audit the
   `update()` rebuild trigger. Future audits should include a row per
   StateField for "rebuilds on bare `setEditMode`".

2. **`alerts.ts` `isNestedBlockquote` 3-level nesting bug** (codex
   review, v0.2.9). The first-pass guard only walked PARENTS; lezer-
   markdown nests `Blockquote` per `>` level so the OUTER quote of
   `> > [!NOTE]` had no `Blockquote` ancestor and slipped through. Fix:
   walk DESCENDANTS too. **Lesson**: matrix risk #5 ("`> [!NOTE]` +
   nested blockquote: alert decoration semantics unclear") was right
   to flag this — the implementation agent caught it correctly under
   review, validating Phase A's "unknown" tagging.

3. **`pendingImagePaste` queue-flush trigger bug** (code-reviewer
   review, v0.2.11). The first cut gated the flush on the bare
   `filePath` transition, which fires for ANY context switch — File >
   Open, sidebar click, quick-open — and would silently insert queued
   image bytes into a freshly opened, unrelated file. Data-loss-
   adjacent. Fix: `pendingDrainArmed` ref set ONLY by save-driven
   entry points; non-save opens call `clearPendingImages()`. **Lesson**:
   any "queue + flush on event" feature needs to enumerate the FULL
   set of transitions that fire the flush trigger, not just the happy
   one. Add a test that models the gated-vs-armed paths side-by-side.

4. **CSP `connect-src` boundary** (v0.2.10). Phase A didn't catch that
   the new HTML image inliner needed renderer-side `fetch` against
   `durumi-asset://`, which the original CSP blocked (no `connect-src`
   declared). Solved cleanly by extending the existing `img-src
   durumi-asset:` boundary to `connect-src` too — but a future feature
   that needs renderer→main fetches against any other protocol would
   hit the same wall. **Lesson**: CSP review belongs in any feature
   that touches a custom protocol from a NEW DOM API.

### Should be closed in a v0.2.12 (recommendation)

**None.** Every "real" gap is either closed or has an honest deferral
rationale. The two candidates I considered:

- *Citation `[-@key]` test*: 1-line assertion in
  `tests/editor/citation.test.ts`. Not worth a release-tag-and-publish
  cycle on its own; bundle into the v0.3 references work.

  regression in Live mode. Same logic — bundle into v0.3 if any v0.3
  table change touches Live.

**Recommendation: ship v0.3.0 directly, no v0.2.12.** The cycle has
hit a clean equilibrium: every ⚠/✗ is either ✅, deferred with
rationale, or accepted with rationale. A v0.2.12 would be ceremony
without user-visible value.
